2.1 모듈 개요
────────────────────────────────────────

core/ 디렉토리는 XGif의 저수준 녹화, 인코딩, GPU 가속 컴포넌트를 포함합니다.
wxPython 독립적으로 설계되어 CLI에서도 재사용 가능합니다.


2.2 파일별 상세 분석
────────────────────────────────────────

[version.py — 6줄]
- 용도: 버전 관리 Single Source of Truth
- 상수: APP_NAME, APP_VERSION, EDITOR_VERSION
- 참고: v1.0.3, CI/CD 관행 준수

[defaults.py — 14줄]
- 용도: 공유 기본 설정 상수
- COMMON_DEFAULTS: 14개 기본값 (언어, 캡처 백엔드, 인코더, 코덱, 메모리 한도 등)
- 설계: CLI/UI 단일 진실 소스

[utils.py — 434줄]
- 용도: core 공유 유틸리티 함수
- 주요 함수:
  * get_resource_path(): PyInstaller frozen 앱 지원
  * ensure_system_site_packages(): CuPy 외부 venv 경로 주입
  * calculate_overlay_position(): 위치 계산 (웹캠, 워터마크, 키보드)
  * apply_alpha_blend(): RGBA 블렌딩 (opacity)
  * load_system_font(): 폴리폴백 폰트 로딩
  * parse_resolution(): 유연한 해상도 문자열 파싱
  * validate_resolution(): 해상도 범위 검증
  * safe_delete_timer(): wxPython 타이머 정리
  * safe_rmtree(): Windows 파일 잠금 내성 디렉토리 삭제
  * run_subprocess_silent(): Windows 콘솔 창 억제
- 의존성: PIL, numpy
- 설계 패턴: 폴백 체인, 플랫폼 감지

[capture_backend.py — 535줄]
- 용도: 화면 캡처 추상화 레이어
- 주요 클래스:
  * CaptureBackend (Abstract): 인터페이스
  * DXCamBackend: DXGI Desktop Duplication (GPU 가속)
    - 공유 카메라 패턴 (클래스 레벨 _shared_camera)
    - 인스턴스 추적 (_instance_count)
    - atexit 훅으로 정리
    - 재사용 위한 100ms 워밍업
  * FastGdiBackend: 직접 GDI BitBlt + GetDIBits (DC 재사용)
    - PIL 백업이 있는 빠른 폴백
  * GdiBackend: PIL ImageGrab 래퍼 (레거시)
- 팩토리 함수:
  * create_capture_backend(): 자동/수동 선택
  * test_capture_backend(): 하드웨어 테스트
  * get_available_backends(): 열거
- 스레드 안전성: 공유 카메라 작업에 잠금

[screen_recorder.py — 1,080줄]
- 용도: 메인 화면 녹화 오케스트레이션
- 주요 클래스:
  * ScreenRecorder: 메인 레코더
    - 콜백 기반 이벤트 시스템
    - 스레드 기반 캡처 + 프레임 수집기
    - 공유 메모리 버퍼 (numpy 배열, 스레드 안전)
  * CaptureThread: 워커 스레드
    - 오버레이 렌더링 (커서, 클릭 하이라이트, 워터마크, 키보드)
    - HDR 보정 적용
    - 100프레임마다 성능 프로파일링
    - 타이밍 드리프트 보정
- 녹화 흐름:
  1. start_recording(): 영역/FPS 검증, 버퍼 생성, 캡처+수집 스레드 시작
  2. CaptureThread.run(): 프레임 캡처, 오버레이 적용, 공유 버퍼 기록
  3. 수집 스레드: 버퍼 → 프레임 리스트 복사
  4. stop_recording(): 스레드 조인, 실제 FPS 계산
- HDR 처리: 첫 프레임에서 적응형 감지, 이후 보정 모드 캐시
- 스레드 안전:
  * Threading.Event (stop, pause, shm_event, shm_processed_event)
  * 정리 중 None 역참조 방지를 위한 로컬 변수 캐싱

[gif_encoder.py — 1,329줄]
- 용도: GIF/MP4 인코딩 (FFmpeg)
- 주요 클래스:
  * GifEncoder: 메인 인코더
    - 품질 프리셋 (high/medium/low)
    - H.264/H.265 코덱 선택
    - 인코더 우선순위 (NVENC>QSV>AMF>CPU)
- 인코딩 경로:
  1. Pipe 모드 (선호): 프레임 바이트를 FFmpeg stdin으로 직접 전송
     - 2-pass: palettegen + paletteuse (GIF)
     - 비동기 프레임 쓰기 (데드락 방지)
  2. FFmpeg BMP 모드: pipe 실패 시 폴백
  3. Pillow 모드: FFmpeg 없을 때 폴백
- 주요 기능:
  * GPU 감지: CuPy, pynvml, nvidia-smi
  * 인코더 자동 선택: 하드웨어 인식 H.264/H.265
  * MP4 지원: 오디오 믹싱, yuv420p 제약 (홀수 치수 크롭)
  * 비동기 stderr 처리 (데드락 방지용 별도 스레드)
  * 디스크 공간 체크: 인코딩 전 검증
  * 하드웨어 폴백: GPU 실패 시 CPU 재시도
- 에러 처리:
  * process.kill() (try-finally)
  * stderr 캡처 (진단용)
  * 타임아웃 강제 (600s / 10분)

3.7 capture_overlay.py — 영역 선택 윈도우
────────────────────────────────────────

[CoordinateConverter (DPI 스케일링 유틸)]
- 논리 (wx DIP) <-> 물리 (픽셀) 좌표 변환
- get_screen_at(x, y): 좌표 포함 모니터 감지
- get_dpi_scale(display): PPI -> 96-DPI 스케일 비율
- logical_to_physical(): DPI 스케일 + 모니터 오프셋 적용
- physical_to_logical(): 역 변환
- 멀티 모니터 지원 (각 모니터별 DPI)

[CaptureOverlay (투명 선택 윈도우)]
- 윈도우 속성: FRAME_NO_TASKBAR | STAY_ON_TOP | BORDER_NONE
- Layered Window (Win32 API): RGB(0,0,0) 투명 키
- 내부 사각형: 캡처 영역 테두리 (빨강)
- 외부 핸들: 8 코너 + 4 엣지 (15x15px)
- 하단 배지: FPS/크기/녹화 상태 텍스트
- 투명도: 보통 100%, 녹화 중 30%

[핸들 시스템 (9개)]
  TOP_LEFT(1)     TOP(2)        TOP_RIGHT(3)
  LEFT(8)         [캡처 영역]    RIGHT(4)
  BOTTOM_LEFT(7)  BOTTOM(6)     BOTTOM_RIGHT(5)
  MOVE(9): 테두리 드래그 (전체 이동)
  NONE(0): 영역 외부

[영역 계산]
  padding = handle_size + handle_margin = 15 + 3 = 18px
  border_offset = (border_width + 1) // 2 = (3 + 1) // 2 = 2px
  window_width = capture_width + 2 * padding + 2 * border_offset

[키보드 지원]
  화살표 키: +-1px 이동
  Shift+화살표: +-10px 이동

4.11 에디터 코드 품질 관찰
────────────────────────────────────────

[강점]
  + 잘 조직된 core vs UI 분리
  + 메모리 최적화 (지연 로딩, 압축 저장, 썸네일 캐싱)
  + Non-blocking UI (모든 긴 작업 백그라운드 스레드)
  + GPU 가속 (옵션, CPU 폴백)
  + 다중 백엔드 (pyvips, OpenCV, Pillow — 우아한 성능 저하)
  + 적절한 Lock, Event, wx.CallAfter()
  + 포괄적 로깅
  + 독스트링 풍부

[약점 & 이슈]
  - 에러 복구: 일부 메서드가 로드 실패 시 빈 이미지 반환 (에러 전파 대신)
  - GPU 초기화: 초기 체크 후 재확인 없음 (런타임 변경 가능)
  - 파일 핸들 누수: 일부 Image.open()이 적절히 닫히지 않을 수 있음
  - 트랜잭션/롤백 없음: 배치 작업 중간 실패 시 일부 프레임 수정, undo 불확실
  - 디스크 스왑 미구현: get_swap_path()은 있으나 실제 스왑 없음
  - 썸네일 캐시: 대규모 GIF에서 축적, 보다 공격적 캐시 무효화 필요
  - 동시 작업 불가: ThreadPoolExecutor에 1스레드만 사용
  - async/await 없음: threading + 콜백 사용 (wxPython에는 적절)


4.12 편집 작업 흐름
────────────────────────────────────────

일반적 편집 순서:
  1. GIF/비디오 로드 -> GifDecoder/VideoDecoder -> FrameCollection 생성
  2. 사용자 프레임 선택 -> Canvas 표시 -> PropertiesPanel 업데이트
  3. 사용자 이펙트 적용 -> Worker 스레드 실행 -> UndoManager에 기록
  4. 사용자 저장 -> SaveWorker -> GifEncoder -> 디스크 기록
  5. Undo -> UndoManager.undo() -> 프레임 복원 -> Canvas 갱신

     — 새 설정 추가 시 4곳 동시 수정 필요, 누락 버그 잦음

[권장 실행 순서]:
  Phase 1 (기반) -> Phase 4 (테스트) -> Phase 2 (리팩터링) -> Phase 3 (에디터) -> Phase 5 (성능)
